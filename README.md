# 🦁 ことばであそぼ！しりとり

**幼児（4〜6歳）の語彙育成を目的とした、AI搭載しりとりアプリ。**
遊びながら「ことば」を覚え、拡散的思考力（あるヒントからいろいろ思いつく力）を育てます。

## なぜ作るのか（プロダクトの意義）

単なるしりとりの実装ではなく、**「しりとりを世の中の課題に接続する」**という方針で設計した。

事実調査の結果、しりとりには次のエビデンスがあることがわかった：

- しりとりは**ワーキングメモリ・想起力・拡散的思考力・語彙力**を鍛える。WHO
  2019「認知症リスク低減ガイドライン」も認知トレーニングを推奨している。
- 幼児は**4歳8ヶ月〜5歳4ヶ月**でしりとりが可能になり、使う語が「動物→食べ物」と**カテゴリを拡大**しながら語彙を増やす。
- 既存のしりとりアプリは「音声・CPU対戦」に寄っており、**「幼児の語彙育成に特化し、AIで優しく導く」**方向は差別化の余地がある。

そこで本アプリは、**幼児がひとりでも／親と一緒でも、優しく語彙を増やせる知育プロダクト**を目指した。

## 実装機能

### 必須機能

1. **直前の単語を表示**（大きな文字＋よみ＋「つぎは○から！」）
2. **任意の単語を入力**できる
3. **末尾↔先頭の一致判定**：一致で更新／不一致はやさしくエラー表示
4. **「ん」で終わるとゲーム終了**
5. **既出単語の再入力でゲーム終了**
6. **リセット機能**（ゲーム中・終了後どちらからも）

### 追加機能

- **①
  育成の可視化（履歴）**：使った単語を一覧表示し、**カテゴリ別の語彙数**（どうぶつ／たべもの…）を集計。終了時に「きょうは○こ
  つかえたね！」と成果を提示 → 親が子の成長を実感できる。
- **②
  ヒント（段階スキャフォールド）**：しりとりが鍛える「拡散的思考力」を奪わないよう、**答えではなく"思考の足場"を段階的に開示**する。
  1. カテゴリ示唆（「"こ"から始まる、どうぶつにいるかな？」）
  2. 連想なぞなぞ（AI生成：「白くてふわふわ、"こ"から…なーんだ？」）
  3. 頭2文字だけ開示
  4. 最終手段として具体語を提示
- **③ 読み上げ**：Web Speech
  API（`SpeechSynthesis`）で単語・ヒントを音声読み上げ。字が読めない幼児でも遊べる。
- **④ AIによる言葉判定**：入力語の**よみ・実在性・カテゴリ**をAIが判定（下記）。
- **⑤
  AI対戦（難易度切替）**：子とAIが交互に打つ対戦形式。AIは頭文字から始まる実在語を返す（「ん」で終わらない）。AIが続けられなくなったら子の勝ち。難易度「やさしい／ふつう」を切り替え可能。**AIの手もサーバー側の決定論コードで再検証**してから採用する。

### デザイン（幼児向けUI）

- **大きな文字・やさしい配色・絵文字**で、字が読めない幼児でも直感的に触れる。
- 現在の語・手番（🧒 きみ／🤖
  AI）・「つぎは○から！」を大きく提示し、今なにをすべきかが一目で分かる。
- 使った言葉は**話者アイコン付きで履歴表示**し、終了時に「きょうは○こ
  つかえたね！」と成果を見せる（親向け）。
- **読み上げ**で耳からも学べる。

## 設計思想：AIと決定論の切り分け

しりとりの技術的な難所は「単語のよみ（末尾/先頭のかな）」の取得にある。「林檎」でも「りんご」でも末尾が「ご」だと知るには**読み仮名**が要る。静的辞書は未知語で詰む。

そこで **LLM（Groq）に「単語 →
よみ・実在性・カテゴリ・ヒント」を判定させる**ことでこの難所を解いた。ただし——

> **ゲームの勝敗ルール（末尾一致・「ん」終了・既出）は、AIのゆらぎに委ねず、必ず決定論コードで確定させる。AIは"ゆらぎのある判断"（よみ・実在・カテゴリ・ヒント）だけを担う。**

この切り分けにより、ハルシネーションが勝敗を狂わせることを防いでいる。

APIキーはクライアントに晒せないため、**Denoサーバーが安全なプロキシ**としてAIを呼ぶ。これが「なぜサーバーがあるか」の技術的必然になっている。

```
[ブラウザ] ──入力語──▶ [Deno /api/judge] ──▶ [Groq API]
  勝敗は決定論で確定      GROQ_API_KEY を        Llama等(OpenAI互換)
  (public/shiritori.js)   env から取得（秘匿）    {よみ,実在,カテゴリ}
```

**縮退運転（フォールバック）**：AI障害・APIキー未設定時も、実在チェックをスキップして**末尾一致の基本ルールだけでゲームを続行**する（画面に「AIおやすみちゅう」を表示）。ゲームは止まらない。

## 技術選定・意思決定（自分で選んだこと）

本課題で「実装」以上に大事にしたのは、**作る意義から逆算して、根拠を持って選ぶ**こと。主要な意思決定は次の通り。
思考の主体は自分で、AIは補助です。

1. **課題設定＝「幼児の語彙育成」**：単なるしりとり実装ではなく、まず事実調査（しりとりの脳トレ効果・幼児の語彙発達）を行い、そこから逆算してターゲットと機能を決めた。
2. **技術スタック＝Deno＋素のJS／HTML、Deno
   Deploy**：フレームワークを持ち込まず、軽量でデプロイの容易な構成にした。
3. **「勝敗は決定論／ゆらぎはAI」の切り分け**：ゲームの勝敗（末尾一致・「ん」・既出）は決定論コードで確定し、AIはよみ・実在・カテゴリ・ヒントだけを担う。ハルシネーションが勝敗を壊さない設計。
4. **辞書問題をAIで解く**：静的辞書は未知語で詰むため、LLMに「よみ・実在・カテゴリ」を返させ、漢字・カタカナ入力にも対応した。
5. **AIバックエンドの選定と移行（Claude→Gemini→Groq）**：無料で継続運用できることを重視して比較。最終的に**無料枠のあるGroq（OpenAI互換）**を採用。プロバイダ依存は
   `lib/ai.ts`
   に隔離し、サーバー・フロントを無変更で載せ替えられる設計にしたため、移行は1ファイルで完了した。
6. **ヒントは「段階スキャフォールド」**：答えを渡すとしりとりが鍛える"拡散的思考力"を奪うため、方向→なぞなぞ→頭2文字→答え、と段階的に開示する設計にした。
7. **スコープ判断**：AI対戦は実装。親対戦・全国対戦は通信・マッチング・認証が重く今回の範囲を超えると判断し、意図的に見送った。

## 技術構成

| 項目       | 採用                                               |
| ---------- | -------------------------------------------------- |
| ランタイム | Deno                                               |
| サーバー   | `Deno.serve` ＋ `@std/http` の静的配信             |
| フロント   | 素の HTML / CSS / JavaScript（フレームワークなし） |
| AI         | Groq（`llama-3.3-70b-versatile`・無料枠）          |
| デプロイ   | Deno Deploy                                        |

```
├── main.ts               # サーバー: 静的配信 + /api/judge + /api/hint + /api/opponent
├── lib/ai.ts             # Groq API 呼び出し（OpenAI互換・JSONモード）
├── lib/shiritori_test.ts # 決定論ロジックの単体テスト
├── public/
│   ├── index.html
│   ├── app.js            # ゲーム進行・状態管理・読み上げ
│   ├── shiritori.js      # 決定論ロジック（ブラウザ・Deno 共用）
│   └── style.css
└── deno.json
```

## ローカルで動かす

```bash
# 1. APIキーを設定（AI機能を使う場合。.env に書けば deno task が自動読込）
export GROQ_API_KEY="..."   # https://console.groq.com で無料取得
# 任意: モデルを変えたいとき（既定 llama-3.3-70b-versatile）
# export GROQ_MODEL="..."

# 2. 開発サーバー起動（http://localhost:8000）
deno task dev

# 3. テスト
deno task test
```

APIキーを設定しなくても、縮退運転で基本のしりとりは遊べる。

## デプロイ（Deno Deploy）

1. GitHub のパブリックリポジトリに push する
2. [Deno Deploy](https://deno.com/deploy)
   でプロジェクトを作成し、本リポジトリを連携（エントリポイント：`main.ts`）
3. 環境変数 **`GROQ_API_KEY`** を Deno Deploy の Secret に設定する
4. デプロイされたURLを本READMEの「デプロイURL」に記入

## デプロイについて（既知の問題）

提出時点で、**Deno Deploy
の公開URLにアクセスできない**問題が発生しています。原因の切り分けと対応の記録を残します。

- **症状**：Deno Deploy のデプロイ工程のうち
  `Warm up`（デプロイ後にアプリを起動して疎通確認する健全性チェック）と `Route`
  が失敗し、新しいビルドが公開に反映されない。
- **ローカルでは全機能が正常動作**：`deno task dev`
  で必須6機能＋追加機能（履歴・段階ヒント・読み上げ・AI判定・AI対戦）すべて動作を確認済み。単体テスト（`deno task test`）も通過。
- **調査**：アクセスログ上、warm-up 相当のプローブ `GET /https://example.com`
  に対して静的配信の `serveDir` が **301 リダイレクト**を返しており、Deno Deploy
  側の**オープンリダイレクト検査に抵触した可能性が高い**と推測。少し前まで同じ構成でデプロイ成功していたことから、プラットフォーム側の検査強化とタイミングが重なったと考えている。
- **対応**：`//` を含む不審なパスはリダイレクトさせず 404 を返すよう `main.ts`
  を修正済み（該当コミットあり）。ただし提出時点で公開反映を確認できていない。
- **レビュー方法**：下記「[ローカルで動かす](#ローカルで動かす)」の手順で、審査時にローカルで全機能を再現・確認いただけます。

## AI使用箇所の詳細

本アプリは **Groq（`llama-3.3-70b-versatile`・無料枠、OpenAI互換API）**
を以下の3エンドポイントで使用している（実装は `lib/ai.ts`）。いずれも
`temperature=0` ＋
**JSONモード（`response_format: json_object`）**で、出力のブレを抑えている。

### `POST /api/judge` — 言葉判定

- **入力**：ユーザーが入力した単語（漢字・カタカナ・ひらがな可）
- **AIの役割**：現代仮名遣いの**よみ**、**実在する言葉か**、**幼児向けカテゴリ**を判定
- **使い道**：よみを決定論ロジック（`public/shiritori.js`）に渡して勝敗を確定。実在しない語はやさしく差し戻し（知育）。カテゴリは履歴集計に使用

### `POST /api/hint` — 段階ヒント生成

- **入力**：次の頭文字・ヒントレベル・使用済み単語・カテゴリ
- **AIの役割**：頭文字から始まる幼児向けの語を選び、**答えを言わない連想なぞなぞ**を生成
- **使い道**：レベルに応じて開示（なぞなぞ→頭2文字→具体語）

### `POST /api/opponent` — AI対戦の一手

- **入力**：次の頭文字・使用済み単語・難易度
- **AIの役割**：頭文字から始まり「ん」で終わらない実在語を1つ選ぶ
- **使い道**：サーバーが `public/shiritori.js`
  の決定論ロジックでAIの手を再検証し、正しければ採用。出せなければAIの降参（子の勝ち）

### 開発におけるAI活用

本プロジェクトの設計方針の壁打ち・事実調査・実装・コードレビューに、Claude
Code（Anthropic）を利用した。
